# 业务知识

这个文件沉淀当前项目中可长期复用的领域知识和稳定口径。

## 当前实现口径

- 当前 workflow 原型采用 `Dify Workflow + Ollama/qwen3:8b` 作为实现事实，抽取层已经落地在三份文档的结构化抽取节点上。
- Dify 的职责边界是初审引擎，不承担人工复核留痕和正式审批回写。
- 系统内部的主输出和规则编码口径已经固定，样本真值中的小写值仅用于验收映射。

## 领域术语

- `采购合同`：采购方向供应商约定采购主体、商务条款、履约方式和关键责任条款的正式文本。
- `供应商报价单`：供应商给出的报价和部分商务条款确认文本，是价格与条件的重要参考来源。
- `PO`：采购订单，代表采购方实际下单动作，是最终执行单据。
- `DocumentPackage`：V1 固定输入包，必须同时包含 `contract`、`quotation`、`po` 三类文档。
- `FinalDecision`：V1 固定主结论枚举，仅允许 `PASS`、`REJECT`、`MANUAL_REVIEW`。
- `正式风险`：会直接触发阻断结论的规则命中，等级为 `P1` 或 `P2`。
- `人工复核项`：不直接算正式风险，但需要人工确认的待确认事项，等级为 `MR`。

## 业务规则

- V1 固定支持 8 类正式风险，不新增第 9 类风险。
- V1 固定支持 2 类非风险型人工复核场景，不将其误用为正式风险扩展口。
- 规则优先级为：先正式风险，后人工复核规则；同一字段同时满足两类规则时，以正式风险为准。
- 命中任一正式风险时，初始结论必须是 `REJECT`。
- 未命中正式风险但命中人工复核规则时，初始结论必须是 `MANUAL_REVIEW`。
- 两类规则都未命中时，初始结论才是 `PASS`。
- `REJECT` 在同一轮审查中原则上不直接改为 `PASS`，补充材料后应重新发起一轮审查。

## 数据与文档语义

### 三类文档的业务语义

- `contract`：承载合同编号、签订日期、采购方、供应商、收货地点、币种、税率、付款条件、总金额、交期以及关键履约条款。
- `quotation`：承载报价单编号、出具日期、供应商、报价有效期、币种、税率、付款条件、交期以及报价行项目。
- `po`：承载 PO 编号、下单日期、采购方、供应商、币种、税率、付款条件、交期、收货地点和最终下单行项目。

### 统一对象语义

- `ExtractedDocument`：三类文档统一抽取结果，最少应包含文档编号、文档类型、供应商、采购方、币种、税率、付款条件、交期、收货地点、行项目和合同关键条款。
- `LineItem`：行项目标准结构，核心字段是 `item_code`、`item_name`、`spec_model`、`quantity`、`unit`、`unit_price`、`tax_included`、`delivery_batch`。
- `ReviewFinding`：单条审查发现，区分 `risk`、`manual_review_flag`、`info`，并保留规则编码、证据、建议动作和是否必须人工复核。
- `ReviewReport`：完整审查输出，至少包含主结论、文档概览、字段比对结果、正式风险、人工复核事项、人工复核区和审查留痕。

### 固定比对字段

- 单据头部字段：`supplier_name`、`buyer_name`、`currency`、`tax_rate`、`payment_terms`、`delivery_date`、`receiver_location`
- 行项目字段：`line_items.item_code`、`line_items.item_name`、`line_items.spec_model`、`line_items.quantity`、`line_items.unit_price`、`line_items.delivery_batch`

## 风险与人工复核口径

### 8 类正式风险

- `R001 供应商不一致`：合同、报价单、PO 的 `supplier_name` 实质不一致，且不能通过主体映射证明为同一主体。
- `R002 物料不一致`：已匹配行项目的 `item_code`、`item_name` 或 `spec_model` 实质不一致。
- `R003 数量不一致`：已匹配行项目的 `quantity` 不一致，且没有明确分批下单说明。
- `R004 单价不一致`：已匹配行项目的 `unit_price` 不一致。
- `R005 交期不一致`：`delivery_date` 或可解析交付时间不一致，且不属于语义等价表达。
- `R006 付款条件不一致`：`payment_terms` 不一致，且没有有效审批依据。
- `R007 币种 / 税率不一致`：`currency` 或 `tax_rate` 不一致。
- `R008 缺少关键条款`：合同缺失 `验收条款`、`质量/质保条款`、`违约责任条款` 中任一项。

### 2 类人工复核项

- `MR001 近似物料名待确认`：`item_name` 近似但 `item_code` 与 `spec_model` 一致，更像别名或不同叫法。
- `MR002 半结构化条款待确认`：合同或报价单中的交期、付款条款是自然语言表达，和 PO 结构化字段语义基本一致但抽取置信度不足。

## 已知边界情况

- 供应商主体存在简称、别名、集团主体映射时，不自动判定一致，需要人工确认。
- 物料名称近似但编码与规格一致时，不自动判定为风险，需要人工确认是否同一物料。
- 自然语言交期、分批交付、分段付款等条款可能语义等价，但 V1 不做激进自动归一。
- 合同正文缺失关键条款但附件可能补充时，应进入人工确认，而不是直接以模型猜测补全。
- 无法唯一匹配行项目时，不能以模糊匹配结果直接给出正式风险结论。

## 当前工作假设

- 项目默认服务中文采购业务场景，报告、界面和审查话术以中文为主。
- 行项目匹配默认以 `item_code` 为主匹配键，以 `item_name + spec_model` 为辅。
- RAG 主要用于规则依据、条款模板和自然语言条款标准化辅助，不直接替代规则引擎。
- V1 是业务型 Agent 样板项目，强调可解释与可复核，不追求“单大模型直接给最终结论”的黑盒实现。
